# DEMO — 15 分鐘導覽腳本

給不熟悉 agentic AI 的人看，目標不是說服他們「這比直接問 ChatGPT 厲害」
——誠實跑過真實對照測試後（見 [`practice/stage12/note.md`](practice/stage12/note.md)
的「誠實的自我檢討」），這個結論站不住腳。目標是讓他們看懂**多 agent
互相扮演角色的機制怎麼運作**：persona 怎麼訪談模擬用戶、怎麼各自從單一
角度評分同一個提案、一場會議怎麼從「假設好幾個可能的需求方向」收斂到
「只留一個站得住腳的」再往下發想解法。看完能自己講出這套機制實際上在
做什麼，而不是只有「感覺比較厲害」。

不需要 API key、不需要架伺服器——`demo/sample-run/` 底下的檔案都是
用公開範例設定（`personas.example.yaml`／`users.example.yaml`／
`company.example.md`，全部虛構）真實跑出來的，直接雙擊開啟即可。

## 開場（1 分鐘）——先講規則，不要先講技術

一句話講完整個練習專案在做什麼：

> 「輸入一個題目，系統先假設幾個可能的需求方向、各自訪談模擬用戶驗證，
> 收斂成一個站得住腳的問題之後，換一組跟這家公司實際能力有關、背景各自
> 不同的 AI persona 各自獨立提案，三位評審各自只從單一角度打分數，選出
> 總分最高的做成原型、再找模擬用戶跟『直接問 LLM』的答案做誠實對照。
> 這整套流程叫 agentic AI——等一下不是要證明它比較聰明，是讓你看清楚
> 這個機制實際上在做什麼。」

不要在開場解釋 LangGraph／Send fan-out／checkpointer 這些技術細節，
那是給懂技術的人事後問才展開的。

## 第一步：看對比表（3 分鐘）

打開 `demo/sample-run/replay.html`，畫面最上方就是對比表，逐行念過去：

| 維度 | 這張表在說什麼 |
|---|---|
| 真實搜尋依據 | Baseline 可能編造來源；agent 端贏家提案的引用來自真實搜尋結果 |
| BMC 完整度 | 兩邊都填滿，但 agent 是被程式強制驗證過結構完整，不是模型自稱填滿 |
| 動態組隊（扣著公司能力衍生職能） | 不是隨便給幾個職稱，是先讀這家公司的定位描述衍生出真的有、彼此不重複的職能，再各自獨立提案 |
| 候選需求方向覆蓋率 | 系統先假設好幾個可能的需求方向、各自訪談驗證，只挑一個往下走——這裡看得到假設了幾個、選了哪個、其餘的訪談證據還在不在 |
| idea 多樣性 | Baseline 只有一個答案，連「多樣性」這個概念都不存在；agent 端有好幾個獨立發想、可以量測彼此距離的觀點 |
| DFV 結構化評分覆蓋 | 三位評審各自只從單一角度（顧客需求性/技術可行性/商業存續性）評分，不是一個 agent 給一個綜合分數 |
| 成本 | 誠實揭露：agent 流程貴很多，這是用結構換來的，不是免費的午餐 |

**誠實的提醒**：這張表證明的是「agent 流程做了更多結構化的事」，不是
「agent 流程的點子比較好」——兩者不是同一件事，這個區分是這次導覽最
重要的認知調整。

## 第二步：播放會議回放（8 分鐘）

往下滑到「會議實況回放」，按播放（可以調到「快速」）。挑幾個時間點暫停講解：

1. **`desk_research_hypothesize_jobs`**：讀這家公司的定位描述做五力/
   趨勢分析，假設幾個可能的需求方向——注意這一步刻意不講任何解法，
   只描述「這群人在什麼情境下卡在哪」
2. **`interview_turn`**：點開幾則問答——問法是回溯式的「你上一次真的
   下定決心處理這件事是什麼時候」「當時考慮過什麼替代方案又為什麼
   放棄」「哪個瞬間讓你卡關」，這是 JTBD 的 switch 訪談技巧，訪談對象
   不知道背後是在幫產品做研究
3. **`select_job_and_define_problem`**：點開 `why_selected`——為什麼是
   這個需求方向被選中，依據是哪些具體的訪談證據，不是空泛判斷；沒被
   選中的方向也還留著，可以往回找它們的訪談逐字稿
4. **`generate_one_persona_for_domain`**：每一位參與者的「職能」是先從
   這家公司實際定位描述衍生出來的，不是隨便湊的頭銜——點開看某一位的
   `domain` 是不是真的跟公司描述對得上
5. **`draft_idea`**：每人獨立提案，看得到各自設計的 Business Model
   Canvas——彼此完全沒看過對方的想法
6. **`ask_question`（人類介入點）**：這是唯一真人可以插手的地方，可以
   對任一提案追問，跳過就直接進評分
7. **`dfv_score`**：三位評審對同一個提案給出的分數可能互相矛盾——這是
   刻意設計的，不是 bug，商業存續性評審覺得不可行、顧客需求性評審覺得
   很想要，兩者同時成立才是誠實的評分
8. **`evaluate_with_agents`**：模擬使用者不知道哪個方案花了更多功夫做
   出來，同時對 agent 方案跟 baseline 給意見＋分數——這是整場最接近
   「有沒有用」的證據，但樣本只有 2-3 人，不要過度解讀

## 第三步：點開原型（2 分鐘）

`demo/sample-run/` 底下命名為 `*-winner.html` 的檔案——這是總分最高的
那個提案，AI agent 自己生成文案、渲染成的真實 landing page，不是文字
描述。只有一個，不是每個人各生成一份。

## 第四步：翻報告（1 分鐘）

`demo/sample-run/final-report.md` 是整場會議的完整記錄：候選需求方向
與各自的訪談證據（含落選的）、選定依據、每位參與者的提案與 BMC、DFV
評分、收斂結果、原型連結、人類問答、baseline 對照、最終評估者的誠實
對照分數。這份報告本身就是「這不是黑盒子」的證據——每一個結論都能回溯
到具體的訪談逐字稿或評分紀錄。

## 誠實地說：這場 demo 沒有證明什麼

單看一場會議，agent 流程比 baseline 貴了幾十倍——如果只看這個數字，
容易覺得「不划算」，但更誠實的問題其實是：**貴了這麼多，換到的東西
是不是真的比較好？** 真實跑測的答案並不篤定：

- 量測到的 idea 多樣性經常低於「同一主題下真的不同」的校準基準值，
  有幾次甚至接近「同一個 idea 換句話說」的程度——結構上是平行獨立發想，
  但量出來的距離沒有穩定證明這件事
- 拿一份真實的內部腦力激盪記錄做對照，發現「開放式問題框架＋刻意跨
  領域的角色設定」這兩件事，直接找一個人＋一次 LLM 呼叫也做得到，不
  一定需要整套多 agent 流程

這場 demo 真正展示的是**結構化的過程**：假設被驗證、決策有依據可回溯、
評分有結構、對照是誠實的（不是自己人球員兼裁判）——這些是真的，跟「這
比人類/比直接問 LLM 更會想點子」是兩件不同的事。如果你的目標是學這套
機制怎麼搭、什麼地方會踩坑，這場 demo 值得看；如果你的目標是找一個能
保證產出更好點子的工具，這場 demo 不是那個證明。

## 收尾：問觀眾，不要幫他們下結論

問觀眾：「你覺得這整套機制實際上做了什麼，值不值得？」不要引導到
「比 ChatGPT 好」這個方向，比較誠實的討論題是：

1. **這套機制驗證了什麼、沒驗證什麼**——訪談證據、評分依據都可以追溯，
   但「回溯得到」不等於「品質比較好」
2. **多樣性是真的還是量出來的假象**——平行獨立發想不保證真的不同，
   量化指標才看得出來，而且量出來的結果不一定漂亮
3. **貴出來的成本，換到的是流程的可信度還是產出的品質**——這個專案
   自己的真實測試比較支持前者

## 給懂技術的人：怎麼重新產生這份樣本

`demo/sample-run/` 是用範例設定（`personas.example.yaml`／
`users.example.yaml`／`company.example.md`，全部虛構）跑 stage13
真實產生的，不會動到你自己的真實設定檔（程式在這些檔案不存在時本來
就會 fallback 到 `*.example.*`）：

```bash
cd practice/stage13
mkdir -p /tmp/demo_run
../.venv/bin/python3 run_worker.py start --run-dir /tmp/demo_run \
  --thread demo-sample --topic "你想測試的主題" --example-config
# 跑到人類介入點會自動停下來，state.json 的 status 會變成 paused
../.venv/bin/python3 run_worker.py resume --run-dir /tmp/demo_run \
  --thread demo-sample --action skip --example-config
../.venv/bin/python3 build_replay.py /tmp/demo_run/events.jsonl \
  /tmp/demo_run/stage13-run-<跑完印出的 timestamp>.json \
  /tmp/demo_run/replay.html
```

跑完後把 `events.jsonl`／`reports/demo-sample-final-report.md`／
`prototypes/demo-sample-winner.html`／`stage13-run-*.json`／
`replay.html` 複製進 `demo/sample-run/`（依序改名成
`events.jsonl`／`final-report.md`／`prototype-winner.html`／
`run-data.json`／`replay.html`），取代舊檔案即可。全程約 $0.5-1、
幾分鐘，不是免費的操作。

**已知的小落差**：這樣直接用 CLI 跑（不透過 `server.py`）沒有
`meta.json` 可以在 resume 階段回填主題，最終存檔的 `topic` 欄位會退回
預設字串，不是你在 `--topic` 打的那個——內容本身仍然正確（第一階段
就已經照你給的主題做完調查與訪談），只有標題文字對不上，不影響這份
範例的可用性。
